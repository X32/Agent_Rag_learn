**问题 1（RAG 流程）**

请简述 RAG 的"检索"和"生成"两个阶段分别做了什么？为什么需要结合两者而不是只用 LLM？

检索是将用户输入内容进行embeding后从数据库里进行检索，

生成是将检索出的内容放到大模型提示词后进行输出答案，


* LLM 知识有截止时间，无法获取最新信息
* LLM 可能产生幻觉（hallucination）
* RAG 可以让 LLM 基于**私有数据**回答，且答案可追溯来源

**问题 2（RAG 流程）**

在 RAG 系统中，如果检索到的文档与问题不相关，会导致什么问题？有什么方法可以缓解这个问题？


### 一、检索到不相关文档会导致什么问题？

#### 1️⃣ **幻觉（Hallucination）**

LLM 会基于检索到的内容生成答案，但如果内容是错误的或不相关的，LLM 仍然会"一本正经地胡说八道"。

```
用户问：SpeakCube 项目的负责人是谁？

检索到的文档：[某篇无关的技术博客，提到张三]

LLM 回答：SpeakCube 项目的负责人是张三。❌（错误但看似可信）
```

#### 2️⃣ **Garbage In, Garbage Out（垃圾进，垃圾出）**

RAG 的核心假设是"检索到的文档是有用的"。如果这个假设被打破，整个系统就失效了。

```
RAG 流程：
用户问题 → 检索 → [不相关文档] → LLM → [错误答案]
                      ↑
              这里错了，后面全错
```

#### 3️⃣ **可信度下降**

用户发现几次错误答案后，就会失去对系统的信任。

---

### 二、缓解方法详解

#### 方法 1️⃣：**设置相似度阈值**

```python
# 伪代码示例
results = vector_db.search(query, top_k=10)

# 过滤掉相似度低于阈值的结果
threshold = 0.7
filtered_results = [r for r in results if r.similarity >= threshold]

if not filtered_results:
    return "抱歉，我没有找到相关信息。"  # 诚实回答
```

**原理：** 如果检索结果和查询的相似度太低，说明不相关，直接丢弃。

**优点：** 简单有效

**缺点：** 阈值需要调优，太高会漏掉相关信息

---

#### 方法 2️⃣：**重排序（Re-ranking）**

```
第一步：用向量检索快速召回 100 条候选（速度快，但精度一般）
                ↓
第二步：用 Cross-Encoder 重排序这 100 条（速度慢，但精度高）
                ↓
第三步：取 Top 5 送给 LLM
```

**为什么需要 Re-ranker？**

* 向量检索是 **双塔模型** ，查询和文档分别编码后计算相似度
* Cross-Encoder 是 **单塔模型** ，查询和文档一起编码，能更好判断相关性

**常用模型：** BGE-Reranker、Cohere Rerank、Jina Reranker

---

#### 方法 3️⃣：**混合检索（Hybrid Search）**

```
向量检索：捕捉语义相似度
     ↓
      合并
     ↓
BM25：捕捉关键词匹配
```

**为什么有效？**

* 向量检索：能找到"语义相关但用词不同"的文档
* BM25：能确保"关键词匹配"的文档不被遗漏

**例子：**

```
用户问："怎么安装依赖？"

向量检索可能找到："如何添加依赖包"（语义相近，但没有"安装"这个词）
BM25 可能找到："npm install 安装依赖"（关键词匹配）

混合检索：两者都要
```

---

#### 方法 4️⃣：**查询改写（Query Rewriting）**

```
原始查询："项目怎么跑？"
    ↓ 改写
新查询："项目 运行 启动 执行 部署"
    ↓ 检索
更相关的结果
```

**常见方法：**

* **扩展** ：添加同义词、相关词
* **简化** ：去除冗余词汇
* **多查询** ：生成多个变体分别检索

---

#### 方法 5️⃣：**LLM 自我判断**

让 LLM 自己判断检索到的文档是否有用：

```python
prompt = f"""
问题：{query}
检索到的文档：{retrieved_docs}

请判断上述文档是否与问题相关？如果相关，请基于文档回答。
如果不相关，请直接说"未找到相关信息"。
"""
```

### 三、方法对比总结

| 方法         | 复杂度 | 效果       | 推荐优先级     |
| ------------ | ------ | ---------- | -------------- |
| 相似度阈值   | ⭐     | ⭐⭐⭐     | 高（必做）     |
| Re-ranking   | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | 高（效果最好） |
| 混合检索     | ⭐⭐   | ⭐⭐⭐⭐   | 中             |
| 查询改写     | ⭐⭐   | ⭐⭐⭐     | 中             |
| LLM 自我判断 | ⭐     | ⭐⭐       | 低（作为补充） |

### 四、实际建议

如果你刚搭建 RAG 系统，按这个顺序优化：

```
1. 先加相似度阈值过滤（最简单）
         ↓
2. 调优 Embedding 模型和切分策略
         ↓
3. 加上 Re-ranking（提升最明显）
         ↓
4. 需要的话再加混合检索
```

**问题 3（文本切分）**

常见的文本切分策略有哪些？请至少说出 2 种，并说明它们各自的适用场景。

**补充/纠正：**

| 切分策略        | 适用场景                               |
| --------------- | -------------------------------------- |
| 固定长度切分    | 简单快速，但可能切断语义               |
| 按句子/段落切分 | 保持语义完整性，适合文档结构清晰的场景 |
| 按语义切分      | 语义连贯性最好，但计算成本高           |
| 递归字符切分    | 平衡效率和语义，适合代码/混合内容      |

**问题 4（文本切分）**

什么是"重叠窗口"（overlap）？在文本切分时为什么要设置 overlap？

因为切分可能会将语义切割，从而不完整，检索的时候只返回部分内容


* ✅ Overlap 是相邻 chunk 之间保留的重复文本
* ✅ 目的：防止关键信息被切分后丢失上下文，保证语义完整性

**问题 5（Embedding）**

Embedding 向量的相似度计算中，余弦相似度（Cosine Similarity）和欧氏距离（Euclidean Distance）有什么区别？RAG 中更常用哪个？


* ✅ 余弦相似度：衡量方向一致性，不受向量长度影响
* ✅ 欧氏距离：衡量绝对距离
* ✅ RAG 更常用 **余弦相似度** （Embedding 通常归一化处理）
