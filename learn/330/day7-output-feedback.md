```python
# RAG 核心流程伪代码
# 1. 加载文档
documents = load_documents("file.txt")

# 2. 文本切分
chunks = split_text(documents, chunk_size=512, chunk_overlap=50)

# 3. 向量化
embeddings = [get_embedding(chunk) for chunk in chunks]

# 4. 存储
store = {embedding: chunk for embedding, chunk in zip(embeddings, chunks)}

# 5. 检索
query_embedding = get_embedding(user_query)
similar_chunks = find_most_similar(query_embedding, store, top_k=3)

# 6. 生成
prompt = build_prompt(user_query, similar_chunks)
answer = llm.generate(prompt)
```

### 1.2 回顾 Day 3-4：文本切分

**核心概念回顾**：

| 概念          | 定义                    | 为什么重要                       |
| :------------ | :---------------------- | :------------------------------- |
| chunk_size    | 每个文本块的大小        | 太大：信息冗余；太小：语义不完整 |
| chunk_overlap | 块之间的重叠部分        | 避免关键信息被切分到两块         |
| 切分策略      | 固定长度/句子/语义/递归 | 不同策略适合不同场景             |

**实验结果回顾**：

```
chunk_size 对比实验结果：
- 128:  检索快但语义不完整
- 256:  平衡
- 512:  推荐起点
- 1024: 语义完整但可能引入噪声
```

**自我检查**：

- [ ] 为什么要切分？不切分行不行？
- [ ] chunk_overlap 的作用是什么？
- [ ] 防止关键信息被切分
- [ ] 如何选择合适的 chunk_size？


| 问题           | 答案                                          |
| -------------- | --------------------------------------------- |
| 为什么要切分？ | 为了精准检索、节省上下文、提高 Embedding 质量 |
| 能不切分吗？   | 短文档可以，长文档基本不行                    |
| 切分多大合适？ | 通常 200-1000 token，根据场景调整             |

| 文档类型          | 推荐 chunk_size  | 理由                 |
| ----------------- | ---------------- | -------------------- |
| FAQ/问答对        | 不切分或 500-800 | 本身就是一个完整语义 |
| 技术文档/API 文档 | 300-500          | 一个函数/接口说明    |
| 普通文章/博客     | 500-800          | 1-2 个段落           |
| 书籍/长文档       | 800-1000         | 保持章节完整性       |
| 代码文件          | 按函数/类切分    | 语义边界清晰         |
| 对话记录          | 300-500          | 一个话题范围         |




```
你的文档是什么类型？
    │
    ├─ 很短（<500 字） → 不切分
    │
    ├─ 结构化强（FAQ/问答） → 按条目切分
    │
    ├─ 技术文档/API → 300-500
    │
    ├─ 普通文章 → 500-800
    │
    └─ 书籍/长文档 → 800-1000，按章节切分

确定后，用 10-20 个测试问题验证，调整到召回率最高
```


| 考虑因素 | 建议                    |
| -------- | ----------------------- |
| 首要原则 | 保持语义完整性          |
| 起始值   | 500 token + 100 overlap |
| 验证方法 | 用测试问题测召回率      |
| 避免     | 在句子中间硬切          |

### 1.3 回顾 Day 5-6：Embedding

**核心概念回顾**：

| 概念              | 定义             | 要点                         |
| :---------------- | :--------------- | :--------------------------- |
| Embedding         | 文本→向量的映射 | 语义相似的文本向量距离近     |
| 向量维度          | 向量的长度       | 常见：768、1024、1536、3072  |
| Cosine Similarity | 余弦相似度       | 衡量方向相似度，范围 [-1, 1] |

**模型对比回顾**：

| 模型                   | 维度 | 特点     | 适用场景       |
| :--------------------- | :--- | :------- | :------------- |
| text-embedding-3-small | 1536 | 性价比高 | 原型、成本敏感 |
| text-embedding-3-large | 3072 | 精度高   | 生产环境       |
| BGE-M3                 | 1024 | 开源免费 | 本地部署       |

### 3.2 常见理解错误自查

在阅读以下陈述时，判断对错并解释：

1. "Embedding 向量的每个维度都有明确的语义含义"

   - **答案**：错。维度是隐式特征，人类难以直接解释
2. "chunk_size 越大越好，因为能保留更多上下文"

   - **答案**：错。太大可能引入噪声，影响检索精度
3. "Cosine Similarity 的值越接近 1 越好"

   - **答案**：不一定。取决于具体场景
4. "RAG 系统只需要向量化一次，之后可以直接复用"

   - **答案**：对，但新增文档需要重新向量化
5. "OpenAI 的 Embedding 模型支持中文"

   - **答案**：对，text-embedding-3 系列支持多语言
